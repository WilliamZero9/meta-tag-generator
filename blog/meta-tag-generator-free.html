What Meta Tags Actually Do

Meta tags are metadata -- instructions to non-human readers (search engines, social scrapers, browser parsers) about the content of a page. There are four buckets that matter, and one that mostly does not.

The Four Buckets That Matter

  • Classic HTML meta: <title>, meta description, viewport, charset, robots. These are the tags search engines read for ranking signals and listing previews. Google uses title and description in SERP results roughly 70% of the time -- the rest of the time it rewrites them based on the page content, and the rewrite is usually worse than what you wrote.
  • Open Graph (OG): og:title, og:description, og:image, og:url, og:type, og:site_name, og:locale. These control how the page previews on Facebook, LinkedIn, Slack, Discord, iMessage, WhatsApp, Signal, and roughly every modern messaging platform that is not Twitter/X. One spec drives most of your distribution.
  • Twitter Cards: twitter:card, twitter:title, twitter:description, twitter:image. X reads OG first and falls back to its own Twitter Card tags -- you need both if you want predictable rendering across the Twitter web preview, embeds, and the X for iOS/Android app.
  • Canonical and structured data: <link rel="canonical"> and JSON-LD blocks (Schema.org types like Article, Product, FAQPage). Canonical tells search engines which URL is the authoritative version when the same content exists at multiple addresses. JSON-LD is how you qualify for rich results in Google -- star ratings, FAQ accordions, article bylines, breadcrumb trails.

The Bucket That Mostly Does Not Matter

Meta keywords. Google stopped using the keywords tag for ranking in 2009. Bing stopped shortly after. Including it does not hurt, but it also does not help -- do not spend time on it. Any free meta tag generator that asks you to fill in a fifteen-item keyword list is optimizing for a ranking signal that has not existed for most of the current internet.

The Five Mistakes That Repeat in Every Audit

Across the landing pages and blog posts audited for this guide, the same five mistakes show up in more than half of them. They are all fixable in under ten minutes each. They are also all invisible until you test the preview explicitly -- which is the real argument for using a meta tag generator rather than hand-typing tags from memory.

1. Wrong Open Graph Image Size

The Facebook documentation says og:image must be at least 200x200. What it does not say loudly is that anything under 1200x630 will render as the small-square "summary" card on most platforms, not the large landscape preview that drives clicks. Twitter's summary_large_image card wants the same roughly 1.91:1 ratio. LinkedIn defers to OG and crops aggressively outside that ratio. The safe target is 1200x630 JPEG or PNG under 5 MB. The small-summary variant uses 1200x1200 square.

Open Graph Image Specs

The Dimensions That Actually Work

Card Type Dimensions Max File Size
Large landscape (default) 1200 x 630 5 MB
Square summary 1200 x 1200 5 MB
Twitter summary_large_image 1200 x 628 (min 300 x 157) 5 MB
LinkedIn (follows OG) 1200 x 627 5 MB

Save the image with an absolute HTTPS URL in the og:image tag. Relative URLs fail on half of scrapers -- the crawler does not know which domain to resolve against.

2. Missing og:type

The og:type tag tells a scraper what kind of page it is looking at -- a website, an article, a product, a video, a book, a music album. Leaving it out defaults to website, which is fine for landing pages and wrong for everything else. An article with og:type=article unlocks additional tags (article:published_time, article:author, article:section) that feed into LinkedIn's publisher preview and Facebook's Instant Articles. A product page with og:type=product unlocks pricing and availability. Use the right type, then use the type-specific tags.

3. Truncated Meta Descriptions

Google truncates meta descriptions at roughly 155 characters on desktop, 120 on mobile. Past those limits, the tail of your description becomes an ellipsis. A meta description that runs to 300 characters is not "more content" -- it is a partial CTA with the punchline cut off. The same applies to og:description (roughly 200 characters before truncation on Facebook, 140 on X) and to og:title (60 characters before truncation, most strict on X and Slack).

Length budget: title 50-60 chars, description 140-155 chars, og:description 150-200 chars. The free meta tag generator flags overruns before you copy the HTML -- one of the few places a tool beats hand-coding.

4. No twitter:card Declaration

X reads OG tags first, but if you do not declare twitter:card it falls back to rendering as a plain link with no preview image. summary_large_image is almost always what you want -- it produces the wide landscape card. summary produces a small square card with text beside it. Leaving the tag out entirely means X picks for you, and it picks badly. Declare it explicitly.

5. Stale or Wrong Canonical

A canonical tag pointing at the wrong URL, or at a URL that redirects, or at a URL that no longer exists, is worse than no canonical at all -- it tells search engines your page is a duplicate of the wrong address and they should index that one instead. Common causes: CMS migrations that leave the old slug in the template, staging URLs that leak into production, mixed HTTP/HTTPS canonical across the same domain. Every meta tag audit should verify the canonical URL returns a 200 on its own and matches the page's actual public URL.

How to Write a Meta Description That Earns the Click

A meta description is not a summary. It is a pitch. Google treats it as optional for ranking, but it is the copy the user reads before deciding whether to click your result over the fourteen other results on the page. The tags that matter for SEO rank you -- the meta description is what converts a ranked page into a visit.

The pattern that works in 2026: open with the problem the page solves, then the specific thing the page delivers, then the proof of specificity. All in under 155 characters.

Meta Description Pattern

Problem. Delivery. Specificity.

[Pain point the reader is feeling, one short clause.] [What this page gives them, concrete.] [The specific hook -- number, year, free, tool name.]

Example: "Open Graph previews rendering gray on Slack? Generate perfect OG, Twitter Card, and canonical tags in 10 seconds. Free tool, no sign-up." -- 136 characters, opens with recognizable pain, names the deliverable, ends with two concrete specifics (speed + cost).

Open Graph Image Size: The Full Reference

The phrase open graph image size gets typed into search bars a hundred times a day because the answer keeps almost-but-not-quite changing. Here is the 2026 reference, with the edge cases the official docs skip.

  • Ratio: 1.91:1 landscape for the default large card. 1:1 square for the small summary card.
  • Minimum dimensions: 600x315 is the floor below which most scrapers downgrade to a small card. 1200x630 is the safe target.
  • Maximum file size: 5 MB for Facebook and LinkedIn, 5 MB for Twitter/X. Most images should be well under 500 KB to render reliably on slow connections.
  • Format: PNG or JPEG. WebP and AVIF are supported by most modern scrapers but still trip legacy crawlers -- stick with PNG or JPEG for maximum reach.
  • Safe area: Keep logos and essential text within the central 1200x600 region. Some platforms crop the top/bottom 15 pixels. LinkedIn crops aggressively on mobile.
  • Alt text: og:image:alt is read by screen readers and some crawlers. Include it. One descriptive sentence.

Twitter Card Generator: What Changed After X

X (formerly Twitter) still reads Twitter Card tags. Despite the rebrand and the API turbulence, the underlying spec has not changed -- twitter:card, twitter:title, twitter:description, twitter:image all still work. What changed is fallback behavior: if Twitter Card tags are missing but OG tags are present, X renders a card using the OG data. If both are missing, it shows an unstyled link. The practical implication: you can usually skip Twitter-specific tags and let OG do the work -- except when you want to control the card type (summary vs. summary_large_image) explicitly, or when you want different copy for X than for LinkedIn. In those cases, declare the full Twitter Card block.

When to diverge: If your OG description is written for LinkedIn (professional, longer, value-prop forward) and X needs something punchier, write a separate twitter:description. Otherwise let OG tags do double duty -- fewer places for drift to creep in.

How to Validate Meta Tags Before You Ship

Writing the tags is half the work. Verifying they render correctly is the other half. Three tools cover the entire preview landscape -- use all three before calling a page done.

  1. opengraph.xyz -- paste any URL, see exactly how the OG tags render across Facebook, LinkedIn, Slack, Discord, iMessage, and a generic preview. Free, no sign-up. The fastest way to catch a wrong image or missing tag.
  2. Twitter Card Validator (cards-dev.twitter.com/validator) -- verifies how X renders your page. Note that X's validator sometimes caches aggressively; if you updated a tag and the preview has not refreshed, wait thirty minutes and revalidate.
  3. LinkedIn Post Inspector (linkedin.com/post-inspector) -- LinkedIn caches OG data for seven days by default, which means a wrong image can haunt you through the critical launch week. The Post Inspector forces a re-crawl. Use it every time you change an OG tag.

A fourth tool worth knowing: Google's Rich Results Test (search.google.com/test/rich-results) checks whether your JSON-LD structured data qualifies for rich results in Google SERPs. It will not tell you anything about OG tags -- use it alongside the three above, not instead of them.

When to Hardcode Meta Tags vs. Use a Generator

A fair question: if meta tags are fifteen lines of HTML, why use a tool at all? Three answers, each a different context.

  • Hardcode when: the site is a single landing page, the tags rarely change, and the developer who owns the template has a current mental model of all five buckets. For indie SaaS landing pages, most marketing sites, and single-page portfolio sites, the right answer is to write the meta block once, validate it in all three tools above, and never touch it again.
  • Use a generator when: the page is the tenth one you are shipping this week, you do not remember whether og:image:width is measured in pixels or percent, and the cognitive cost of looking up the tag names exceeds the cost of typing the URL into a tool. This is also true when someone non-technical on the team is writing meta tags -- a generator with validation built in prevents the invisible bugs described above.
  • Generate dynamically when: the site is a CMS, blog, or e-commerce store where the title, description, and OG image change per page. Here you want meta tag generation baked into the template layer -- Next.js metadata API, Nuxt's useHead, Astro's <Head>, or whatever your framework exposes. A one-off generator is useful for auditing the output of the dynamic system, not for replacing it.

The Common Traps Nobody Warns You About

  1. Slack aggressively caches OG tags. If you post a link, realize the preview is wrong, fix the tags, and re-share the link in the same channel, Slack keeps the old preview for up to an hour. Test in a private channel first. Or append a ?v=2 query string after an update to force a re-fetch.
  2. Single-page apps need server-rendered meta. Client-side meta tag injection via React Helmet works for Google but fails for most social scrapers, which do not execute JavaScript. If you are shipping an SPA, the meta tags must be in the initial server response -- either via SSR, static pre-rendering, or a serverless function that injects tags before delivery.
  3. og:url should match canonical. When they diverge, Facebook and LinkedIn attribute shares to the og:url, but search engines use canonical. The result is split distribution analytics and inconsistent attribution. Point them at the same URL.
  4. Image URLs must be HTTPS in 2026. Scrapers increasingly reject http:// image URLs entirely. If your OG image is served from an older CDN that has not moved to HTTPS, move it before shipping.
  5. Query parameters strip in previews. Sharing https://site.com/post?utm=campaign causes some scrapers to fetch the URL without the query string, resulting in different meta tags than you expected. If your meta tags depend on the path rather than the query string, this is fine. If you personalize meta tags by query string, test every variant.
